## 先看结论

部署不是把一个文件“放到网上”这么简单，而是把下面几层连接起来：

```text
用户浏览器
  ↓
Cloudflare：负责域名解析、外层 HTTPS 和基础防护
  ↓
腾讯云服务器：公网 IP 为 114.132.180.195
  ↓
Nginx：接收 80/443 端口的访问，并转发给应用
  ↓
Next.js：真正运行高校筛选系统，监听 127.0.0.1:3000
  ↓
SQLite：保存学校、项目、用户、申请和材料数据
```

本次部署已经完成：

- `https://shuqi.fun/login` 可以访问；
- `https://www.shuqi.fun/login` 可以访问；
- HTTP 会自动跳转到 HTTPS；
- Cloudflare 到源服务器也使用加密连接；
- 本地数据已迁移到服务器；
- 程序会开机自动启动，异常退出后自动重启；
- 旧网站的根域名和 `www` 记录已被替换；
- `y.shuqi.fun` 保持原样，没有修改。

## 1. 先理解部署里的几个角色

### 1.1 域名是什么

域名可以理解为网站的“门牌号”。

用户不会记住：

```text
114.132.180.195
```

但很容易记住：

```text
shuqi.fun
```

DNS 的工作，就是告诉浏览器：

```text
shuqi.fun 对应哪一台服务器？
```

### 1.2 Cloudflare 是什么

Cloudflare 位于用户和服务器之间，可以理解为“门口保安”。

它主要负责：

- 管理域名 DNS；
- 给用户提供 HTTPS；
- 隐藏部分源站信息；
- 提供基础 DDoS 防护；
- 将访问转发到腾讯云服务器。

### 1.3 Nginx 是什么

Nginx 可以理解为服务器里的“前台接待”。

它负责：

- 接收标准网站端口 `80` 和 `443`；
- 处理 HTTPS 证书；
- 把请求转发给 Next.js；
- 记录访问日志；
- 在未来统一管理多个网站。

### 1.4 Next.js 是什么

Next.js 是高校筛选系统真正的程序。

它会处理：

- 登录；
- 学校和项目查询；
- 筛选规则；
- 客户资料；
- 申请进度；
- 文件上传下载；
- SQLite 数据读写。

### 1.5 SQLite 是什么

SQLite 可以理解为一个单文件数据库。

本项目的主要数据库文件是：

```text
app.db
```

这个文件非常重要。代码坏了通常可以重新部署，数据库丢失则可能导致业务数据丢失。

### 1.6 systemd 是什么

systemd 是 Linux 的服务管理器，可以理解为“程序管家”。

它负责：

- 后台启动系统；
- 服务器重启后自动启动；
- 程序异常后自动重启；
- 统一查看日志和运行状态。

## 3. 开始前必须准备什么

### 3.1 账号和权限

至少需要：

- 腾讯云服务器管理权限；
- 宝塔面板登录权限；
- 服务器终端权限；
- 域名 DNS 管理权限；
- Cloudflare 登录权限；
- 本地项目文件访问权限。

### 3.2 服务器端口

云服务器安全组至少需要开放：

| 端口 | 是否公开 | 用途 |
|---|---|---|
| 80 | 是 | HTTP，主要用于跳转 HTTPS |
| 443 | 是 | HTTPS 正式访问 |
| 22 | 建议限制来源 | SSH 管理 |
| 8888 | 建议限制来源 | 宝塔面板 |
| 3000 | 不要公开 | Next.js 内部端口 |

Linux 服务器通常不需要 Windows 远程桌面端口 `3389`。确认没有用途后，应在腾讯云安全组中关闭。

### 3.3 本地先做验证

在本地项目目录执行：

```powershell
cd D:\codex-all\school-syt
npm ci
npm run typecheck
npm test
npm run build
```

如果 `npm ci` 因锁文件问题失败，可以先处理 `package-lock.json`，不要直接带着错误部署。

判断标准：

- 类型检查通过；
- 测试通过；
- 构建通过；
- 本地登录和筛选功能正常。

### 3.4 先备份数据

首次迁移前，至少备份：

```text
data/app.db
data/uploads/
data/keys/app.key
```

其中 `app.key` 用于解密上传材料。数据库还在但密钥丢失时，部分加密材料可能无法恢复。

### 3.5 私钥安全

不要把 SSH 私钥、Cloudflare API Token、管理员密码放进聊天、Git 或公开网盘。

下面这种通常是 SSH 公钥：

```text
ssh-rsa AAAA...
```

真正的私钥通常是 `.pem`、`.key` 或无后缀文件，内容可能以以下文字开头：

```text
-----BEGIN OPENSSH PRIVATE KEY-----
```

私钥不能公开发送。

## 4. 完整部署流程

## 第一步：确认服务器系统

在宝塔终端或 SSH 中执行：

```bash
cat /etc/os-release
uname -m
df -h
```

分别确认：

- Linux 发行版；
- CPU 架构，一般为 `x86_64`；
- 磁盘剩余空间。

本次服务器最终确认是 OpenCloudOS 9.6，而不是 Windows。

## 第二步：检查已有软件

```bash
node -v
npm -v
/www/server/nginx/sbin/nginx -v
systemctl --version
```

如果 Node.js 不符合项目要求，需要安装合适版本。

本次 Node.js 安装到：

```text
/opt/node-v24.13.0-linux-x64
```

并建立固定入口：

```text
/opt/node24
```

固定入口的好处是以后升级 Node.js 时，可以只修改链接，不必修改所有配置文件。

## 第三步：建立标准目录

```bash
mkdir -p /opt/school-syt/releases
mkdir -p /opt/school-syt/shared/data/uploads
mkdir -p /opt/school-syt/shared/data/imports
mkdir -p /opt/school-syt/shared/data/keys
mkdir -p /opt/school-syt/shared/backups
```

目录设计：

```text
/opt/school-syt/
├── current -> releases/某个版本
├── releases/
│   ├── 20260629221148/
│   └── 下一个版本/
├── shared/
│   ├── data/
│   │   ├── app.db
│   │   ├── uploads/
│   │   ├── imports/
│   │   └── keys/app.key
│   └── backups/
```

为什么要分开：

- `releases` 只放代码；
- `shared/data` 永远保存业务数据；
- `current` 指向当前运行版本；
- 更新失败时可以快速切回旧版本。

## 第四步：安全打包本地项目

不要上传这些目录：

```text
node_modules/
.next/
.git/
临时日志
本地缓存
```

首次迁移数据时，需要把 `data` 一起放入加密包。以后只更新代码时，不要再次覆盖服务器的 `shared/data`。

推荐使用 AES-256 加密：

```powershell
tar -czf school-syt.tar.gz .
openssl enc -aes-256-cbc -salt -pbkdf2 `
  -in school-syt.tar.gz `
  -out school-syt.tar.gz.enc
```

上传到服务器后解密：

```bash
openssl enc -d -aes-256-cbc -pbkdf2 \
  -in /www/wwwroot/school-syt.tar.gz.enc \
  -out /tmp/school-syt.tar.gz
```

解密时输入打包密码。密码不要写进命令历史或文档。

## 第五步：解压到新版本目录

```bash
RELEASE=$(date +%Y%m%d%H%M%S)
mkdir -p "/opt/school-syt/releases/$RELEASE"
tar -xzf /tmp/school-syt.tar.gz \
  -C "/opt/school-syt/releases/$RELEASE"
```

每次都创建新版本，不要直接覆盖 `current`。

这样更新失败时，旧代码仍然存在。

## 第六步：迁移首次数据

只在首次部署时执行。

把本地数据复制到：

```text
/opt/school-syt/shared/data/
```

最终至少应存在：

```text
/opt/school-syt/shared/data/app.db
/opt/school-syt/shared/data/keys/app.key
```

如果有上传材料，还应存在：

```text
/opt/school-syt/shared/data/uploads/
```

## 第七步：安装依赖并构建

进入新版本目录：

```bash
cd "/opt/school-syt/releases/$RELEASE"
```

优先执行：

```bash
/opt/node24/bin/npm ci
```

如果国内网络访问 npm 很慢，可以临时指定镜像：

```bash
/opt/node24/bin/npm ci \
  --registry=https://registry.npmmirror.com
```

然后构建：

```bash
/opt/node24/bin/npm run build
```

注意：不要在构建前把 `data` 做成指向项目外部的符号链接。本项目使用 Turbopack，构建时可能因为数据目录位于项目根目录之外而失败。

更稳定的做法是：

- 构建阶段使用项目目录内的临时 `data`；
- 运行阶段通过环境变量指向 `/opt/school-syt/shared/data`。

## 第八步：创建独立运行用户

不要让网站程序长期使用 root 身份运行。

```bash
id school-syt >/dev/null 2>&1 || useradd --system --home /opt/school-syt --shell /sbin/nologin school-syt
chown -R school-syt:school-syt /opt/school-syt
```

这样即使应用出现安全问题，攻击者也不会直接得到完整 root 权限。

## 第九步：执行数据库迁移

环境变量需要指向永久数据目录：

```bash
cd "/opt/school-syt/releases/$RELEASE"

runuser -u school-syt -- env \
  PATH=/opt/node24/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin \
  DATABASE_PATH=/opt/school-syt/shared/data/app.db \
  UPLOAD_DIR=/opt/school-syt/shared/data/uploads \
  IMPORT_DIR=/opt/school-syt/shared/data/imports \
  APP_KEY_PATH=/opt/school-syt/shared/data/keys/app.key \
  /opt/node24/bin/npm run db:migrate
```

数据库迁移前一定要有备份。

## 第十步：切换当前版本

```bash
ln -sfn "/opt/school-syt/releases/$RELEASE" /opt/school-syt/current
chown -h school-syt:school-syt /opt/school-syt/current
```

此时 `current` 就会指向刚刚构建好的版本。

## 第十一步：配置 systemd

创建：

```text
/etc/systemd/system/school-syt.service
```

参考内容：

```ini
[Unit]
Description=School SYT Next.js Application
After=network.target

[Service]
Type=simple
User=school-syt
Group=school-syt
WorkingDirectory=/opt/school-syt/current
Environment=NODE_ENV=production
Environment=PORT=3000
Environment=PATH=/opt/node24/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin
Environment=DATABASE_PATH=/opt/school-syt/shared/data/app.db
Environment=UPLOAD_DIR=/opt/school-syt/shared/data/uploads
Environment=IMPORT_DIR=/opt/school-syt/shared/data/imports
Environment=APP_KEY_PATH=/opt/school-syt/shared/data/keys/app.key
Environment=SESSION_TTL_HOURS=12
Environment=MAX_UPLOAD_MB=20
ExecStart=/opt/node24/bin/npm start
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

加载并启动：

```bash
systemctl daemon-reload
systemctl enable --now school-syt
systemctl status school-syt
```

检查监听端口：

```bash
ss -lntp | grep 3000
```

正确结果应该显示程序监听：

```text
127.0.0.1:3000
```

不要监听 `0.0.0.0:3000`，除非有明确理由并配置了额外防护。

## 第十二步：配置 Nginx

配置文件：

```text
/www/server/panel/vhost/nginx/shuqi.fun.conf
```

参考内容：

```nginx
server {
    listen 80;
    server_name shuqi.fun www.shuqi.fun;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    http2 on;
    server_name shuqi.fun www.shuqi.fun;

    ssl_certificate /www/server/panel/vhost/cert/shuqi.fun/fullchain.pem;
    ssl_certificate_key /www/server/panel/vhost/cert/shuqi.fun/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

每次修改后先检查：

```bash
/www/server/nginx/sbin/nginx -t
```

只有看到配置正确后，才重新加载：

```bash
/www/server/nginx/sbin/nginx -s reload
```

不要跳过 `nginx -t`。配置写错后直接重载，可能导致所有网站无法访问。

## 第十三步：确认真正的 DNS 服务商

不要只看“域名在哪里买的”，要看 Nameserver。

Windows 可以执行：

```powershell
nslookup -type=NS shuqi.fun
```

本次查询到：

```text
cortney.ns.cloudflare.com
valentin.ns.cloudflare.com
```

所以真正的 DNS 管理平台是 Cloudflare，不是阿里云。

## 第十四步：修改 Cloudflare DNS

当前记录：

| 类型 | 名称 | 内容 | 代理 |
|---|---|---|---|
| A | `shuqi.fun` | `114.132.180.195` | 已代理 |
| CNAME | `www.shuqi.fun` | `shuqi.fun` | 已代理 |
| A | `y.shuqi.fun` | 原地址 | 保持不变 |

为什么 `www` 使用 CNAME：

- 根域名 IP 变化时，只需要修改一处；
- `www` 会自动跟随根域名；
- 配置更容易维护。

修改 DNS 前，先记录旧值，以便回滚。

## 第十五步：创建 Cloudflare 源站证书

Cloudflare 控制台路径：

```text
SSL/TLS → 源服务器 → 创建证书
```

选择：

- Cloudflare 生成私钥和 CSR；
- RSA 2048；
- 主机名 `shuqi.fun` 和 `*.shuqi.fun`；
- 有效期按实际需要选择。

把证书保存为：

```text
/www/server/panel/vhost/cert/shuqi.fun/fullchain.pem
```

把私钥保存为：

```text
/www/server/panel/vhost/cert/shuqi.fun/privkey.pem
```

限制私钥权限：

```bash
chmod 600 /www/server/panel/vhost/cert/shuqi.fun/privkey.pem
```

Cloudflare 源站证书主要用于 Cloudflare 到服务器之间的连接。保持 DNS 橙色云朵代理开启，不要把它当成普通浏览器直接访问的公开证书。

## 第十六步：启用严格 HTTPS

Cloudflare 控制台：

```text
SSL/TLS → 概述 → 配置 → 完整（严格）
```

不要长期使用“灵活”模式。

推荐链路：

```text
浏览器 --HTTPS--> Cloudflare --HTTPS 严格验证--> Nginx
```

## 第十七步：最终验收

检查根域名：

```bash
curl -I https://shuqi.fun/login
```

检查 `www`：

```bash
curl -I https://www.shuqi.fun/login
```

检查 HTTP 跳转：

```bash
curl -I http://shuqi.fun/login
```

预期结果：

| 检查项 | 预期结果 |
|---|---|
| 根域名 HTTPS | 200 |
| `www` HTTPS | 200 |
| HTTP | 301 跳转 HTTPS |
| 登录页标题 | 高校筛查与申请管理 |
| systemd | active (running) |
| Nginx 配置 | syntax is ok / test is successful |

本次实际验收结果：

```text
root_https=200
www_https=200
http=301 redirect=https://shuqi.fun/login
origin_https=200
```

## 5. 本次实际遇到的问题

### 5.1 SSH 端口开放，但连接立即断开

现象：

- `22` 端口能连通；
- SSH 握手后服务器主动关闭连接。

处理：

- 没有继续盲目重试；
- 使用已经登录的宝塔网页终端完成部署；
- SSH 修复留到部署完成后单独处理。

经验：

> 端口开放不等于服务一定可登录。部署任务应优先保证有一条可靠的管理通道。

### 5.2 原以为是 Windows，实际是 Linux

原因：腾讯云安全组里存在 `3389` 规则，看起来像 Windows 服务器。

处理：通过终端运行系统检测，确认是 OpenCloudOS 9.6。

经验：

> 不要只根据安全组端口判断系统，必须执行 `cat /etc/os-release`。

### 5.3 `package-lock.json` 导致依赖安装失败

现象：服务器上的锁文件无法正常使用，`npm ci` 失败。

临时处理：

- 移开异常锁文件；
- 使用国内 npm 镜像安装；
- 完成构建。

长期正确做法：

- 本地重新生成并验证 `package-lock.json`；
- 将正确锁文件纳入版本管理；
- 服务器优先使用 `npm ci`。

### 5.4 Turbopack 不允许构建时读取项目外部数据链接

现象：数据目录指向 `/opt/school-syt/shared/data` 后，构建失败。

原因：构建工具不允许部分文件通过符号链接逃出项目根目录。

处理：

- 构建时使用项目目录内的临时数据；
- 运行时通过绝对环境变量读取永久数据；
- 数据仍然保存在 `shared/data`。

### 5.5 在阿里云控制台找不到正确 DNS

原因：域名可能在阿里购买，但 Nameserver 已经切到 Cloudflare。

处理：查询 NS 记录，确认 Cloudflare 才是真正的 DNS 管理平台。

### 5.6 Cloudflare API 令牌只有读取权限

现象：可以查询 DNS，但修改时报鉴权错误。

处理：使用已经登录的 Cloudflare 控制台进行修改。

经验：

> API Token 必须按最小权限配置。读取令牌不能修改 DNS，这是正常的安全限制。

### 5.7 Cloudflare 已开启“完整”，但源站没有证书

风险：Cloudflare 会尝试通过 HTTPS 连接服务器，如果 Nginx 没有正确证书，可能出现 525 或 526。

处理：

- 创建 Cloudflare Origin CA 证书；
- 安装到 Nginx；
- 确认源站 HTTPS 返回 200；
- 再切换为“完整（严格）”。

## 6. 常见报错怎么判断

| 现象 | 最可能原因 | 先检查什么 |
|---|---|---|
| 域名打不开 | DNS 未生效或记录错误 | Cloudflare DNS、NS 记录 |
| 502 Bad Gateway | Next.js 没运行或端口错误 | `systemctl status school-syt` |
| 525 SSL Handshake Failed | Cloudflare 连不上源站 HTTPS | Nginx 443、证书、私钥 |
| 526 Invalid SSL Certificate | 严格模式下证书验证失败 | 域名、证书有效期、证书路径 |
| 无限跳转 | Cloudflare 模式与 Nginx 跳转冲突 | 是否误用“灵活”模式 |
| 登录后又回登录页 | Cookie、Host 或 HTTPS 不一致 | 域名访问、Nginx 转发头 |
| 数据库只读 | 文件权限错误 | `ls -l app.db`、目录所有者 |
| 上传失败 | `UPLOAD_DIR`、权限或大小限制 | 环境变量、`MAX_UPLOAD_MB` |
| 重启后网站消失 | 没设置开机自启 | `systemctl is-enabled school-syt` |
| Nginx 重载失败 | 配置语法错误 | `nginx -t` |

### 6.1 502 的排查顺序

```bash
systemctl status school-syt
journalctl -u school-syt -n 100 --no-pager
ss -lntp | grep 3000
curl -I http://127.0.0.1:3000/login
```

如果内部 `127.0.0.1:3000` 都打不开，先修应用，不要先改 DNS。

### 6.2 HTTPS 的排查顺序

```bash
/www/server/nginx/sbin/nginx -t
ss -lntp | grep 443
openssl x509 \
  -in /www/server/panel/vhost/cert/shuqi.fun/fullchain.pem \
  -noout -subject -issuer -dates
```

然后检查 Cloudflare 是否为“完整（严格）”。

### 6.3 数据权限排查

```bash
ls -ld /opt/school-syt/shared/data
ls -l /opt/school-syt/shared/data/app.db
ls -l /opt/school-syt/shared/data/keys/app.key
```

需要时修复：

```bash
chown -R school-syt:school-syt /opt/school-syt/shared/data
chmod 600 /opt/school-syt/shared/data/keys/app.key
```

## 7. 以后更新代码怎么做

更新代码和首次部署不同：

> 更新代码时不要覆盖 `/opt/school-syt/shared/data`。

标准流程：

1. 本地修改代码；
2. 本地执行类型检查、测试和构建；
3. 备份服务器数据；
4. 打包代码，不包含 `data`；
5. 上传服务器；
6. 解压到新的 `releases/时间戳`；
7. 安装依赖并构建；
8. 需要时执行数据库迁移；
9. 把 `current` 指向新版本；
10. 重启服务；
11. 验收登录、筛选和数据；
12. 保留至少一个旧版本用于回滚。

关键命令：

```bash
systemctl restart school-syt
systemctl status school-syt
journalctl -u school-syt -n 100 --no-pager
```

## 8. 出问题如何回滚

假设旧版本目录是：

```text
/opt/school-syt/releases/20260629221148
```

回滚：

```bash
ln -sfn /opt/school-syt/releases/20260629221148 /opt/school-syt/current
systemctl restart school-syt
systemctl status school-syt
```

然后检查：

```bash
curl -I http://127.0.0.1:3000/login
curl -I https://shuqi.fun/login
```

如果新版本执行了不可逆数据库迁移，只回滚代码可能不够，还需要恢复数据库备份。

所以数据库迁移前必须备份。

## 9. 数据如何备份

项目自带：

```bash
npm run backup
```

备份脚本会处理：

- SQLite 数据库；
- 上传材料；
- 文件加密密钥 `app.key`。

生产环境执行时，要带上正确环境变量：

```bash
cd /opt/school-syt/current

runuser -u school-syt -- env \
  PATH=/opt/node24/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin \
  DATABASE_PATH=/opt/school-syt/shared/data/app.db \
  UPLOAD_DIR=/opt/school-syt/shared/data/uploads \
  APP_KEY_PATH=/opt/school-syt/shared/data/keys/app.key \
  /opt/node24/bin/npm run backup
```

备份完成后，把生成的 `backups/时间戳` 移到：

```text
/opt/school-syt/shared/backups/
```

建议策略：

| 频率 | 保留时间 |
|---|---|
| 每日备份 | 7 天 |
| 每周备份 | 4 周 |
| 每月备份 | 6 至 12 个月 |

只放在同一台服务器不算真正安全。至少再同步一份到：

- 腾讯云 COS；
- 另一台服务器；
- 加密后的本地硬盘。

恢复前先停止写入，并先保留当前损坏现场，不要直接覆盖。

## 11. 安全检查清单

### 必须立即处理

- 修改默认管理员账号 `admin / admin`；
- 使用至少 12 位强密码；
- 不在聊天和文档中保存密码；
- 确认 Cloudflare 为“完整（严格）”；
- 确认数据库和 `app.key` 已备份。

### 宝塔终端修改管理员密码

先确认当前代码和永久数据库路径，再在宝塔终端执行：

```bash
cd /opt/school-syt/current
read -rsp "请输入 admin 新密码: " ADMIN_PASSWORD; echo
printf '%s\n' "$ADMIN_PASSWORD" | runuser -u school-syt -- env \
  PATH=/opt/node24/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin \
  DATABASE_PATH=/opt/school-syt/shared/data/app.db \
  /opt/node24/bin/npm run admin:password -- admin
unset ADMIN_PASSWORD
```

密码不会写入命令历史；脚本会更新密码哈希、记录审计并清除该账号旧会话。执行前应先按备份章节备份 `/opt/school-syt/shared/data/app.db`。

确认服务实际读取永久数据库：

```bash
PID=$(systemctl show -p MainPID --value school-syt)
tr '\0' '\n' < "/proc/$PID/environ" | grep '^DATABASE_PATH='
```

预期输出：

```text
DATABASE_PATH=/opt/school-syt/shared/data/app.db
```

### 建议尽快处理

- 宝塔 `8888` 端口只允许自己的固定 IP；
- SSH `22` 端口只允许可信 IP；
- 关闭无用的 `3389` 规则；
- 不对公网开放 `3000`；
- 定期更新系统安全补丁；
- 定期检查磁盘和错误日志；
- 配置自动异地备份；
- 为宝塔开启双因素认证；
- 修复并验证 SSH 密钥登录，关闭密码登录。

### 不要做的事

- 不要直接删除 `shared/data`；
- 不要把私钥提交到 Git；
- 不要修改 Nginx 后跳过语法检查；
- 不要在没有备份时执行数据库迁移；
- 不要为了临时排错把所有端口永久开放；
- 不要长期使用 Cloudflare“灵活”模式。

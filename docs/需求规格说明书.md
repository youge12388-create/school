# 高校筛选与申请管理系统 — 需求规格说明书

> 版本：v0.1（首版）  
> 最后更新：2026-06-29  
> 面向用户：在华留学顾问团队

---

## 一、项目背景

留学顾问在日常工作中需要大量查询高校和专业信息、根据学生条件筛选合适的项目、管理客户档案、跟踪申请进度、保存申请材料。这些工作原本分散在 Excel 表格、微信聊天记录、本地文件夹中，信息容易丢失、版本混乱、协作困难。

本系统旨在提供一个本地运行的一体化 Web 应用，将**高校知识库**、**项目筛选**、**客户管理**、**申请跟踪**、**材料管理**整合到一个工具中，让顾问团队可以高效完成日常工作，同时为未来的服务器部署和企业微信集成预留扩展边界。

### 核心原则

- 首版只做本地单机，保留多账号和角色，避免未来扩展时重写
- 不确定的数据不推断为"符合"或"不符合"，统一显示"数据库无相关信息"
- 不接入外部 AI，专业匹配使用可维护的同义词组，结果稳定且可解释
- 企业微信作为独立适配层，不混入核心业务表

## 三、功能需求

### 3.1 登录与账号

- 账号只能由管理员创建
- 基于 Session Token 的 Cookie 认证（school_syt_session）
- 密码使用 scrypt 哈希存储，要求至少 10 位
- 三次失败登录记录审计日志
- 会话有效期可配置（默认 12 小时）
- 本地开发测试账号 dmin / admin

### 3.2 仪表盘

登录后的首页，展示关键统计数据与待办事项：

- 学校总数、项目总数、客户总数、待复核项目数
- 近 7 天需跟进的客户列表
- 近 30 天截止的项目列表
- 待补充材料的申请列表
- 最近 10 条审计日志

### 3.3 学校管理

- 学校列表：展示名称、省份、城市、QS 排名、合作评级、CSCA 状态
- 支持按名称、省份、城市搜索
- 学校详情：基本信息、所有项目列表（含专业、费用、语言要求等）
- 从筛选结果进入学校详情时，默认只显示相关的项目；可切换查看全部项目

### 3.4 项目管理

- 项目列表：支持按学历类型、教学语言、复核状态筛选，按关键词搜索
- 项目详情：完整展示费用结构、语言要求、GPA 要求、年龄限制、截止日期、奖学金信息、住宿信息等
- 项目数据通过 Excel 导入或手动录入

### 3.5 数据导入

#### Excel 批量导入

- 同时上传学校汇总表和项目汇总表两张 Excel
- 自动校验表头和工作表格式
- 每行同时保存原始 JSON 和结构化字段
- 无法确认的字段使用 UNKNOWN，项目标记为 NEEDS_REVIEW
- 预览阶段比较已有数据，显示新增、修改、完全重复和人工数据冲突
- 确认后在单个 SQLite 事务中更新学校、项目、专业和导入批次

#### 手动录入

- 仅学校中文名必填，其余字段可留空
- 同名学校自动关联
- 使用相同解析器生成结构化字段
- 缺失项目类型或教学语言时保存为 UNKNOWN 并标记待复核
- 手工项目设置 manually_verified，后续 Excel 预览显示冲突且确认时不会覆盖
- 已有相同学校、项目类型和教学语言的有效项目会阻止重复创建

### 3.6 项目筛选

核心功能，根据客户条件匹配推荐项目。

#### 筛选条件

| 条件 | 说明 |
|------|------|
| 申请学历 | 本科 / 硕士 / 博士 / 长期进修 / 短期项目 |
| 教学语言 | 中文 / 英文 / 法文 |
| 目标专业 | 基于关键词和同义词组匹配 |
| 入学年份 | 文本包含检查 |
| 首年预算 | 首年费用上限判断 |
| 是否有 CSCA | 仅本科项目需要 |
| GPA | 仅在计分制一致时自动比较 |
| HSK 级别和分数 | 仅中文项目需要 |
| 英语成绩 | IELTS / TOEFL / Duolingo 任一项达标 |
| 年龄 | 最小/最大年龄范围 |
| 国籍 | 文本规则识别 |
| 地区偏好 | 省/市级别名称匹配 |
| 是否需要奖学金 | 有奖学金信息即通过 |
| 是否需要住宿 | 文本规则识别是否提供住宿 |
| 是否需要导师接收函 | 文本规则识别（支持部分要求） |
| 截止日期模式 | 开放中 / 已过期 / 未知 / 全部 |

#### 匹配逻辑

- 规则引擎计算（非 AI）：每个条件产生一条证据（PASS / NEED / UNKNOWN / FAIL）
- 计分公式：通过项数×20 - 不通过×100 - 需补充×25 - 未知×8 + 合作评级×3 + QS 排名加成 + 复核状态加成
- 排序优先级：已过期靠后 → 匹配度 → 未知数量 → 分数
- 导师接收函状态：REQUIRED / PARTIAL_REQUIRED / NOT_REQUIRED / UNKNOWN

#### 结果展示

- 分组显示：明确符合 / 需补充条件 / 数据不足 / 明确不符合
- "明确不符合"默认收起
- 每个项目卡片展示所有匹配证据
- 点击可进入学校详情（携带筛选上下文）
- 支持保存为推荐方案

### 3.7 客户管理

- 客户列表：支持按负责人、签约状态、录取状态筛选和搜索
- 客户详情：基本信息、跟进记录、申请列表、上传材料、推荐方案
- 新建客户：填写基本信息和筛选条件
- 核心字段：姓名、国籍、电话、邮箱、微信、当前学历、GPA、语言成绩、目标专业/学历、预算等

### 3.8 申请管理

- 申请列表：展示所有申请及其状态
- 申请详情：展示完整状态变更时间线
- 状态机：材料准备 → 已提交 → 审核中 → 补件 → 录取 / 拒绝 → 签证办理 → 已入学 / 关闭
- 每次状态变更必须填写原因并写入 application_events

### 3.9 材料管理

- 上传：文件类型白名单（PDF/JPEG/PNG/DOCX/XLSX），MIME 类型和扩展名双重校验
- 存储：随机内部文件名 + AES-256-GCM 逐文件加密 + SHA-256 完整性校验
- 下载：必须有效会话，写入审计日志
- 支持关联客户或具体申请

### 3.10 审计日志

记录以下操作：

- 登录成功/失败、退出
- 账号创建/停用/角色变更
- 客户创建/更新/归档
- 跟进记录创建
- 申请状态变更
- 文件上传/下载
- 推荐方案创建
- 数据导入

审计表只有写入和查询入口，不支持更新和删除。

### 3.11 安全管理

- 页面说明文件加密机制、密钥管理、备份要求
- 建议启用 Windows BitLocker
- 密钥与数据分开保管
- 生产环境需配置 HTTPS

## 六、系统架构

`
浏览器 ──→ Next.js App Router（页面 / Route Handler / Server Action）
                │
                ▼
         src/lib 业务逻辑层
         ├── auth/audit/http（基础能力）
         ├── import-service/excel-import/program-parser（数据导入）
         ├── matcher/queries（筛选引擎）
         ├── file-crypto（文件加密）
                │
                ▼
         SQLite 本地数据库  +  data/uploads/ 加密材料文件
`

### 关键设计决策

- **页面只负责展示和表单入口**，复杂业务逻辑放在 src/lib
- **屏幕引擎不使用 AI**，使用业务规则 + 同义词组，结果稳定可解释
- **不明确的数据不推断**，统一使用 UNKNOWN / "数据库无相关信息"
- **Drizzle 负责 schema 和简单查询**，复杂联表和关键事务使用原生 SQLite
- **上传材料不入公开目录**，下载必须走受控 API
- **企业微信在第二阶段接入**，作为独立适配层

## 七、技术栈详解（学习参考）

### 7.1 前端框架：Next.js 16 + React 19

**用了什么：**
- App Router（基于文件系统的路由）
- Server Components（默认情况下所有组件都是服务端组件）
- Server Actions（服务端表单处理，ccount-actions.ts、ctions.ts）
- Route Handlers（API 路由，src/app/api/）
- 布局系统（layout.tsx，(workspace)/layout.tsx 做登录态保护）
- 
ext/headers（读取 Cookie）、
ext/navigation（redirect）

**为什么这样用：**
- 这是一个本地工具型应用，页面以服务端渲染为主，不需要复杂的客户端状态管理
- App Router 的布局系统天然适合"登录前/登录后"两套布局
- Route Handler + Server Action 覆盖了所有 CUD 操作，无需额外搭建 API 服务

**值得学习的点：**
- 如何用 (workspace) 路由组做登录保护
- Server Action 和 Route Handler 的职责边界
- 服务端组件中直接操作数据库（无需 API 层）

### 7.2 数据库：SQLite + Drizzle ORM

**用了什么：**
- 
ode:sqlite（Node.js 24 内置的 SQLite 绑定，无需第三方 C++ 编译）
- drizzle-orm（TypeScript 优先的 ORM）
- drizzle-orm/sqlite-proxy（适配器，让 Drizzle 可以使用自定义数据库连接）
- drizzle-kit（自动生成可审查的 SQL 迁移文件）
- WAL 模式（提升并发读取性能）

**为什么这样用：**
- 本地应用不需要安装数据库服务，SQLite 是最合适的选择
- 
ode:sqlite 是 Node.js 内置模块，零依赖
- Drizzle 提供类型安全的查询构建器，同时不对底层做过多抽象
- 复杂查询（多表联查、条件动态拼接）直接写原生 SQL，更清晰

**值得学习的点：**
- drizzle-orm/sqlite-proxy 的适配机制：只需要实现一个 execute(sql, params, method) 函数
- Drizzle schema 定义中的 uniqueIndex、index、eferences、enum 用法
- 原生 SQLite 和 Drizzle 的混合使用策略
- 迁移文件的管理方式（drizzle/ 目录）

### 7.3 认证系统

**用了什么：**
- 基于随机 Token 的 Session 方案（非 JWT）
- Cookie：school_syt_session，httpOnly + sameSite:strict
- 密码哈希：scrypt（Node.js crypto 内置）
- 权限检查：equireUser() / equireRole()

**为什么这样用：**
- 本地应用不需要 JWT 的无状态优势，Session 方案可以主动控制会话失效
- andomBytes(32).toString("base64url") 生成 256 位随机 token，哈希后存入数据库
- Cookie 的 httpOnly 防止 XSS 窃取，sameSite:strict 防止 CSRF

**值得学习的点：**
- Token 的生成、哈希、存储、验证全流程
- Cookie 的安全属性配置
- 	imingSafeEqual 防止时序攻击（密码验证）
- equireUser() 中如何使用原生 SQLite 做两表联查（sessions + users）

### 7.4 文件加密

**用了什么：**
- es-256-gcm（认证加密，同时提供机密性和完整性）
- 每次加密使用随机 12 字节 IV
- SHA-256 做额外完整性校验（解密后比对）

**为什么这样用：**
- 学生材料包含护照、成绩单等敏感信息，必须加密存储
- AES-256-GCM 是认证加密模式，同时防窃取和防篡改
- 密钥自动生成，保存在 data/keys/app.key，权限设为 600

**值得学习的点：**
- GCM 模式的工作原理和优势（内置认证标签）
- IV（初始化向量）的随机生成和存储方式
- SHA-256 checksum 作为额外的完整性保护层
- 密钥的生命周期管理（生成、存储、权限）

### 7.5 筛选引擎（业务核心）

**用了什么：**
- 纯规则引擎（无 AI）
- 正则表达式文本解析
- 可维护的同义词组匹配
- 数量化评分和排序

**为什么这样用：**
- 留学筛选是决策支持工具，结果需要可解释、可审计
- AI 匹配结果不稳定，规则引擎每次运行结果一致
- 同义词组存储在数据库中，顾问可以自行维护

**值得学习的点：**
- 如何设计 Evidence 数据结构（label / level / detail 三元组）
- 正则表达式如何从中文文本中提取结构化信息（费用、语言、年龄等）
- 排序算法如何平衡多个维度（匹配度、未知项、合作评级、QS 排名）
- majorMatches() 函数：先精确匹配 → 回退到同义词组匹配
- getSupervisorAcceptanceStatus()：分段解析 → 分段判断 → 综合结论

## 九、核心流程一览

### 数据导入流程

`
Excel 上传 → 校验表头 → 解析每行 → 预览（对比已有数据）
                                                 ↓
                                   管理员确认 → SQLite 事务写入
                                    学校 + 项目 + 专业 + 审计
`

### 筛选流程

`
读取全部项目中非归档数据
        ↓
根据条件逐一匹配每个项目
        ↓
生成证据链（每条证据含 label/level/detail）
        ↓
计算匹配度和评分 → 排序 → 分组展示
`

### 文件安全流

`
上传：校验 MIME/扩展名 → 随机命名 → AES-256-GCM 加密 → 存储
下载：校验会话 → 解密 → 返回原始文件 → 记录审计日志
`
